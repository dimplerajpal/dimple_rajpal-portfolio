Procurement SaaS Workflow Design B2B Enterprise
01

RFQ → Quote → PO Workflow Optimization

Redesigning the end-to-end procurement transaction flow to eliminate submission errors, enforce compliance rules, and reduce manual corrections by 60%.

🔍 Problem

Enterprise buyers were experiencing high submission failure rates during the RFQ-to-PO journey. Manual corrections were consuming significant ops bandwidth, and incomplete or non-compliant transactions were creating downstream risk for a procurement client managing over $1.2B in spend.

💡 Product Thinking

Mapped the full transaction lifecycle — from RFQ creation to vendor quote submission to PO generation. Identified every point where data inconsistency or missing validations caused failures. Prioritized fixes using MoSCoW: must-have compliance validations before workflow UX improvements.

⚙️ Solution

Architected KPI-driven validation rules at the quote submission layer. Defined field-level validation specs (required fields, format enforcement, business rule triggers). Designed real-time error surfacing in the UI to prevent silent failures. Built structured PRDs covering 50+ validation scenarios aligned to enterprise compliance requirements.

🏗️ System Design Thinking

Designed validation as a layered system: UI-level for immediate feedback, API-level for business rule enforcement, and DB-level for data integrity. Mapped integration touchpoints between RFQ module, vendor portal, and PO generation service. Documented data flows and state transitions in FSDs for engineering.

🧪 QA Strategy
  • Defined 100+ acceptance criteria covering happy path, edge cases, and failure states
  • Edge cases: partial submissions, concurrent editing by multiple users, decimal precision in pricing fields
  • UAT sessions with 10+ enterprise stakeholders to validate against real procurement scenarios
  • Regression suite for all validation rule changes before each release

Access Control Approval Systems Enterprise SaaS
02

Multi-Tier Approval System & Role-Based Access Control

Designing and shipping a configurable, multi-tier approval engine and RBAC framework to enforce procurement governance for 50–200 daily active users.

🔍 Problem

Enterprise procurement clients required strict governance over spend authorization — transactions were being processed without mandatory approvals, creating compliance violations and audit risk. Existing flows had no configurable approval hierarchy or role-specific access boundaries, making it impossible to enforce policy at scale.

💡 Product Thinking

Ran discovery sessions with procurement heads, finance teams, and IT admins to map approval authority at each spend threshold. Translated organizational hierarchy into configurable product logic. Designed RBAC as a cross-cutting concern — roles gate both UI visibility and API-level permissions to prevent privilege escalation.

⚙️ Solution

Architected a multi-tier approval engine where approval chains are dynamically resolved based on transaction type, spend value, and department. Built role definitions covering buyer, vendor, approver, admin, and read-only personas. Designed approval state machine: Draft → Pending Approval → Approved/Rejected → PO Issued, with full audit trail at each transition.

🏗️ System Design Thinking

Modeled the approval engine as a rules-based decision tree: conditions (spend threshold, category, entity) trigger approval tier assignment. Role permissions stored as a matrix mapping (role × resource × action). Documented state transitions, timeout handling, and delegation logic in system flow diagrams for engineering handoff.

🧪 QA Strategy
  • Matrix testing: all role × permission combinations validated (buyer cannot approve own PO, etc.)
  • Edge cases: simultaneous approvals, approver out-of-office delegation, threshold boundary conditions (e.g. ₹99,999 vs ₹1,00,000)
  • Security testing: privilege escalation attempts, API calls with invalid role tokens
  • Facilitated UAT with 5+ procurement managers to validate real-world authority maps

Release Management QA Strategy Defect Prevention
03

Zero Critical Defect Release Strategy

Building and owning a release quality framework that sustained a 95% on-time rate across 100+ weekly production deployments — with zero critical post-deployment defects.

🔍 Problem

A weekly release cadence of 100+ production deployments created constant defect risk. Each cycle involved 30+ defects requiring triage, fix, and re-validation — and with enterprise clients relying on the platform daily, a single critical defect reaching production was unacceptable. The challenge: move fast without breaking things.

💡 Product Thinking

Shifted quality left: acceptance criteria became the first artifact written in every feature ticket — before design, before engineering. By defining "done" precisely upfront, QA and engineering operated from the same contract. Introduced a defect severity classification system (P0–P3) to ensure critical issues were never deferred across release cycles.

⚙️ Solution

Established a structured release gate: features could only advance to staging when all acceptance criteria had corresponding test scenarios documented. Built a defect tracking workflow in Jira with mandatory root-cause fields to enable pattern detection. Owned the final pre-release sign-off, personally validating critical path flows before every deployment.

🏗️ System Design Thinking

Mapped the full defect lifecycle: discovery → triage → assignment → fix → re-test → regression → sign-off. Designed branching logic for critical vs. non-critical defects — P0/P1 blockers halt the release; P2/P3 tracked for next cycle. Maintained a release readiness dashboard in Notion visible to all stakeholders in real time.

🧪 QA Strategy
  • Acceptance criteria written as Given/When/Then scenarios for every user story
  • Mandatory regression suite on all modified modules before production push
  • API contract testing via Postman to catch integration breaks before UI testing
  • SQL validation queries to verify data integrity post-migration and post-deploy
  • 30+ defects resolved per cycle over 95+ consecutive on-time releases

AI-Assisted Product Development & Rapid Prototyping

Reducing engineering load by 80% and prototype-to-deploy time by 60% by embedding LLM-assisted development into the product lifecycle — from PRD to production-grade MVP.

🔍 Problem

Engineering sprint capacity was consistently consumed by prototyping and scaffolding work rather than core feature development. Stakeholder demos required long lead times, slowing down requirement validation. 💡 Product Thinking

Identified that LLMs could compress the prototype-to-staging cycle dramatically when guided by precise, well-structured product specs. Developed a prompting discipline: scoping modules as independent units, defining data models and UI states upfront, and specifying integration contracts before generation. ⚙️ Solution

For ProgressPulse: architected a mobile app with 5 integrated modules including AI progress tracking, photo logging, and timelapse generation.

🏗️ System Design Thinking

StockMate: Designed 4 decoupled modules for independent scaling. Defined database schema, auth strategy, and API contracts before generation. ProgressPulse: Architected background pipeline for image processing, storage strategy for photo logs, and auth flow covering session management. Both products shipped with production-grade technical blueprints validated through LLM-assisted design review.

🧪 QA Strategy
  • LLM-generated code treated as untrusted: all outputs reviewed against acceptance criteria before integration
  • Functional testing on every module boundary before stitching integrations
