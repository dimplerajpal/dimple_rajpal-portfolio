Procurement SaaS Workflow Design B2B Enterprise
01

RFQ → Quote → PO Workflow Optimization

Redesigning the end-to-end procurement transaction flow to eliminate submission errors, enforce compliance rules, and reduce manual corrections by 60%.

🔍 Problem

Enterprise buyers were experiencing high submission failure rates during the RFQ-to-PO journey. Manual corrections were consuming significant ops bandwidth, and incomplete or non-compliant transactions were creating downstream risk for a procurement client managing over $1.2B in spend.

💡 Product Thinking

Mapped the full transaction lifecycle, from RFQ creation to vendor quote submission to PO generation. Identified every point where data inconsistency or missing validations caused failures. Prioritized fixes using MoSCoW: must-have compliance validations before workflow UX improvements.

⚙️ Solution

Architected KPI-driven validation rules at the quote submission layer. Defined field-level validation specs covering 50+ validation scenarios. Designed real-time error surfacing in the UI to prevent silent failures.

🏗️ System Design

Designed validation as a layered system: UI-level for immediate feedback, API-level for business rule enforcement, and DB-level for data integrity. Mapped integration touchpoints between RFQ module, vendor portal, and PO generation service.

🧪 QA Strategy
  • Defined 100+ acceptance criteria covering happy path, edge cases, and failure states
  • Edge cases: partial submissions, concurrent editing, decimal precision in pricing
  • UAT sessions with 10+ enterprise stakeholders validating real procurement scenarios
  • Regression suite for all validation rule changes before each release

Access Control Approval Systems Enterprise SaaS
02

Multi-Tier Approval System & Role-Based Access Control

Designing and shipping a configurable, multi-tier approval engine and RBAC framework to enforce procurement governance for 50–200 daily active users.

🔍 Problem

Enterprise procurement clients required strict governance over spend authorization, transactions were being processed without mandatory approvals, creating compliance violations and audit risk. No configurable approval hierarchy or role-specific access boundaries existed.

💡 Product Thinking

Ran discovery sessions with procurement heads, finance teams, and IT admins to map approval authority at each spend threshold. Designed RBAC as a cross-cutting concern, roles gate both UI visibility and API-level permissions to prevent privilege escalation.

⚙️ Solution

Architected a multi-tier approval engine where approval chains are dynamically resolved based on transaction type, spend value, and department. Designed approval state machine: Draft → Pending → Approved/Rejected → PO Issued, with full audit trail.

🏗️ System Design

Modeled the approval engine as a rules-based decision tree: conditions trigger approval tier assignment. Role permissions stored as a matrix mapping (role × resource × action). Documented state transitions, timeout handling, and delegation logic for engineering.

🧪 QA Strategy
  • Matrix testing: all role × permission combinations validated
  • Edge cases: simultaneous approvals, delegation, threshold boundaries (₹99,999 vs ₹1,00,000)
  • Security testing: privilege escalation attempts, invalid role token API calls
  • UAT with 5+ procurement managers validating real-world authority maps

Release Management QA Strategy Defect Prevention
03

Zero Critical Defect Release Strategy

Building and owning a release quality framework that sustained a 95% on-time rate across 100+ weekly production deployments, with zero critical post-deployment defects.

🔍 Problem

A weekly release cadence of 100+ production deployments created constant defect risk. Each cycle involved 30+ defects requiring triage, fix, and re-validation, and with enterprise clients relying on the platform daily, one critical defect in production was unacceptable.

💡 Product Thinking

Shifted quality left: acceptance criteria became the first artifact written in every feature ticket, before design, before engineering. Introduced a defect severity system (P0–P3) to ensure critical issues were never deferred across release cycles.

⚙️ Solution

Established a structured release gate: features advance to staging only when all acceptance criteria have test scenarios documented. Built a defect tracking workflow with mandatory root-cause fields. Personally owned the final pre-release sign-off.

🏗️ System Design

Mapped the full defect lifecycle: discovery → triage → fix → re-test → regression → sign-off. P0/P1 blockers halt the release; P2/P3 tracked for next cycle. Maintained a release readiness dashboard in Notion visible to all stakeholders in real time.

🧪 QA Strategy
  • Acceptance criteria written as Given/When/Then for every user story
  • Mandatory regression suite on all modified modules before production push
  • API contract testing via Postman to catch integration breaks before UI testing
  • SQL validation queries to verify data integrity post-deploy
  • 30+ defects resolved per cycle over 95+ consecutive on-time releases

AI-Assisted Product Development & Rapid Prototyping

Reducing engineering load by 80% and prototype-to-deploy time by 60% by embedding LLM-assisted development into the product lifecycle, from PRD to production-grade MVP.

Engineering sprint capacity was consistently consumed by prototyping and scaffolding work rather than core feature development. Meanwhile, StockMate and ProgressPulse needed to go from concept to live product, without a dev team.

Identified that LLMs could compress the prototype-to-staging cycle dramatically when guided by precise, well-structured product specs. 🏗️ System Design

🧪 QA Strategy
  • LLM-generated code treated as untrusted: all outputs reviewed against acceptance criteria
  • Functional testing on every module boundary before stitching integrations
  • StockMate: invoice flow, inventory reconciliation, and role access tested with real retail scenarios
  • ProgressPulse: timelapse generation and image pipeline tested with edge-case photo inputs
